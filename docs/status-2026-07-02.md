### Real bugs found (not just planned cleanup)

- **`ParseWaveXPacket` buffer overflow** (`149e7fa`): the function copied a packet's full zero-padded payload region into the caller's destination buffer, ignoring the caller's actual buffer capacity. Any message struct smaller than its packet's padded region (`ErrorMessage`, `SampleLoadMessage`, `SampleMemStatusMessage`, browse-response entries) could be overrun. Reproduced as `*** stack smashing detected ***` the moment exhaustive round-trip tests actually parsed those types back — previously only "was it created" was checked, never "does it parse back correctly."
- **UART `PacketRouter` injection silently disabled** (`c3c7967`): `application_context.cpp` had the real injection call commented out with `// TODO: Fix UART link injection - function not linking properly`. Root cause was a namespace-scoping bug — the hand-rolled `extern` declaration was lexically inside `namespace WaveX`, so it declared the wrong symbol. Production UART traffic was routing through a throwaway `dummy_router` fallback instead of the app's real, owned `PacketRouter`.
- **`CvBus::Flush()` hardcoded to slot 0** (`389349f`): the old CV backend always flushed DAC slot 0 regardless of which voice/group was actually queued. Harmless only because nothing in the tree ever called `QueueVoice`/`Flush` on it. Fixed by construction in the new `Mcp4728Backend`.
- **`VoiceManager` never actually compiled** (`e39ac3a`): a first pass added `voice_manager.hpp` to `CMakeLists.txt`'s source list but never `#include`d it anywhere, so the ARM build never actually parsed it — listing a header as a CMake source doesn't compile it. Caught by grepping for the include before claiming the ARM build verified anything.
- **`Sampler::FeedInputBlock()` never called from production** (`a3fdcfd`): recording is wired up (`OnSampleCtrl` → `StartRec`/`StopRec`), but `Callback()` discards its audio input (`(void)in;`) and never feeds it to the sampler. Recording currently produces silence in production, independent of the real-time-safety fix made to its storage.

### Two roadmap premises that were wrong about the current code

Both corrected in `docs/roadmap.md` rather than silently acted on:

1. Roadmap called `esp_uart_link`/`daisy_uart_link` "legacy" and said to delete them. They're actually the live transport for heartbeat/meter/status/ACK messages, running in parallel with SPI (which only carries browse/wave data). **Left alone** — flagged to you at the time, you confirmed.
2. Roadmap justified enabling `CONFIG_LVGL_PORT_ENABLE_PPA` as low-risk because "we use neither rotation nor triple-partial mode" — but `display_manager.cpp` does call `lv_display_set_rotation(display_, LV_DISPLAY_ROTATION_90)`. The PPA option's own Kconfig text is literally "Enable PPA for screen rotation," so it's likely the *correct* fix, but it also means the "known PPA rotation-mode bugs on P4" caution the roadmap itself raised does apply to us. **Kept enabled** per your call, flagged for hardware verification.

### Item 1 — Output/CV backend seam (`389349f`)

Implements architecture.md §5.3 exactly as specified there (the design was already fully written; this was implementation, not new design):

- `WAVEX_VOICE_OUTPUT_BACKEND` / `WAVEX_CV_BACKEND` / `WAVEX_ANALOG_CV_GROUPS` flags in `hardware_config.h`.
- `StereoMixSink` (real) / `TdmVoiceSink` (compiling stub) in `output_sink.hpp`.
- `CvGroupRouter<Backend, NumGroups>` (template, not virtual — no vtable in the control-tick path) folding voice-indexed CV targets onto physical groups, backed by `Mcp4728Backend` (real I2C) or `Mcp48Backend` (compiling stub).
- New `make daisy-stageb` target + CI step proves both flag-set combinations (Stage A and Stage B) compile — the roadmap's explicit gate for this item.
- 10 new host tests, all HAL-free (no Daisy hardware needed to run them).

**Not wired into the live audio callback.** Today's engine has exactly one playback source; there's nothing to feed the sink/router until item 2's voice manager exists as a real trigger source.

### Item 2 — Voice manager, RAM-resident half (`e39ac3a`)

`VoiceManager`: 8-voice array, oldest-triggered stealing when all 8 are busy, per-voice gain (from velocity)/pan, a pitch hook (hardcoded to 1.0 — note-to-pitch mapping is item 4's job), zero-I/O `Trigger()` (stores a pointer into already-SDRAM-resident sample data, no allocation), `Render()` into item 1's buffer shape. HAL-free, 12 new host tests.

**Split from this item and not started**: "2 concurrent streamed voices with prebuffer admission control." The existing WAV-streaming path (`s_wav`, ring buffer, prebuffer, SD buffer slots) is a separate, still-entirely-singleton subsystem in `audio_engine.cpp` — generalizing it to N concurrent admission-controlled streams is a distinct refactor, not an extension of `VoiceManager`. Scoped out to avoid bundling two large, hardware-unverifiable changes into one commit.

**Not wired into `Callback()`.** `OnNoteOn`/`OnNoteOff` still only drive a leftover test oscillator. There is no note-to-sample mapping policy yet (which MIDI note plays which loaded sample) — that's explicitly item 8's job per the roadmap's own text ("MIDI note path ... → voice manager").

### Item 3 — Sample RAM integration for recording (`a3fdcfd`)

`Sampler::Init()` now takes a `SampleMemMgr&` and preallocates one fixed-capacity extent (30s at the current sample rate) instead of growing a `std::vector` via `push_back` during recording. `audio_engine.cpp`'s init order was fixed so the sample-RAM manager exists before `Sampler` allocates from it.

As noted above, this made *storage* real-time-safe; it did not make recording *functional* — that gap (live audio input never reaches `FeedInputBlock()`) is a separate, deliberately deferred piece of work.

### Why nothing is wired into the live audio callback yet

Every new subsystem this session (`CvGroupRouter`, `VoiceManager`) is built, `#include`d so it's proven to compile for the real ARM target, and thoroughly host-tested — but deliberately not driving `Callback()` yet. I have no physical Daisy hardware in this environment, so I cannot verify actual audio correctness (underruns, glitches, incorrect mixing) on real silicon. Wiring an unverified path into the real-time audio callback would be pure risk for no behavioral change, since there's no real trigger source (MIDI, in item 8) feeding it yet anyway. This wiring happens naturally as items 4 and 8 land.

## Verification methodology (what's actually been checked)

Every commit in this session was verified, from a clean build, before being made:

- **Host tests** (`make test`): 47 shared + 60 Daisy + 79 ESP32 = **186 tests**, 100% passing, run via the vendored GoogleTest with no network access.
- **Daisy firmware**: both `make daisy` (Stage A flags) and `make daisy-stageb` (Stage B flags) build clean from scratch on the real ARM cross-compiler (`arm-none-eabi-gcc`).
- **ESP32 firmware**: built clean from scratch in the `wavex-devcontainer` Docker image (the only environment with ESP-IDF).

What this **does not** prove: actual audio correctness on real hardware (no underrun/glitch testing is possible without a physical Daisy), visual correctness on the real display (PPA rotation), SD card behavior under load, or SPI link timing. Those are called out explicitly wherever they apply, both here and in `CHANGELOG.md`/`docs/roadmap.md`.
